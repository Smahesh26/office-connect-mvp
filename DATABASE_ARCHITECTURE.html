Executive Summary

Office Connect is a comprehensive SaaS platform built on a multi-tenant architecture supporting enterprise-grade operations across multiple business functions:

Platform Capabilities:
  • Multi-organizational support with role-based access control
  • Subscription management with tiered plans and feature modules
  • Customer Relationship Management (CRM) with pipeline tracking
  • Human Resource Management (HRM) with payroll and attendance
  • Inventory & Warehouse management with stock tracking
  • D2C eCommerce platform with stores and orders
  • Point-of-Sale (POS) system integration
  • Invoice generation and GST compliance automation
  • Enterprise accounting with ledger management

Key Metrics

  • Database Type: PostgreSQL (ACID-compliant)
  • Multi-tenancy: Complete data isolation by Organization ID
  • Estimated Database Size: 50-200 GB (depending on transaction volume)
  • Typical Transaction Volume: 10M+ rows in production environments
  • ORM: Prisma (TypeScript-first)

Core Entities

1. Authentication & Authorization

User

Purpose: Platform user account with email-based authentication

  • ID: String (CUID primary key)
  • Email: String (unique)
  • Name: firstName, lastName
  • Auth: passwordHash
  • Role: isPlatformUser (boolean for SUPER_ADMIN)

Relationships: 1:N with OrganizationUser, ProjectMember, File uploads

Role

Purpose: Define role-based access control levels

Enum Values: SUPER_ADMIN | ADMIN | PROJECT_MANAGER | EMPLOYEE | CLIENT

Relationships: 1:N with OrganizationUser

OrganizationUser

Purpose: Join user to organization with specific role (membership)

  • Constraint: Unique (organizationId, userId)
  • Deletion: Cascade on both sides

2. Organization & Subscription

Organization

Purpose: Multi-tenant container for all business data

  • Identity: name, legalName, panNumber, businessType
  • Contact: supportEmail, supportPhone
  • Address: addressLine1, city, state, pincode, country
  • Settlement: Account details for payment processing

Multi-tenant Strategy: All queries filtered by organizationId

Relationships: Central hub to User, Subscription, Project, Contact, Product, etc.

Subscription

Purpose: Track active subscription and billing for organization

  • Status: TRIALING | ACTIVE | PAST_DUE | CANCELED | EXPIRED | SUSPENDED
  • Period: currentPeriodStart, currentPeriodEnd
  • Billing: cancelAtPeriodEnd (boolean)

Relationships: N:1 with Organization and Plan, 1:N with Payment

Plan

Purpose: Define subscription tiers with pricing and feature limits

  • Pricing: price (Decimal), currency, interval (MONTHLY/YEARLY)
  • Limits: userLimit, storageLimit (GB)
  • Features: features[] array of included feature names

Relationships: 1:N with Subscription, 1:N with PlanModule

Module Breakdown

CRM Module - Customer Relationship Management

Contact

Purpose: Master record for customers, vendors, and partners

Type: CUSTOMER | VENDOR | PARTNER

  • Identification: firstName, lastName, companyName
  • Communication: email, phone
  • Compliance: gstNumber, panNumber
  • Address: billingAddress, shippingAddress

Relationships: Linked to Lead, Deal, Order, Invoice, Transaction

Lead

Purpose: Track sales prospects and conversion funnel

  • Status: NEW | QUALIFIED | NURTURING | STALLED | CONVERTED
  • Scoring: score (0-100)
  • Assignment: assignedTo (userId)
  • Lifecycle: isArchived, archivedAt

Relationships: 1:N with Activity, N:1 with Contact

Pipeline & Stage

Purpose: Define deal workflow and stages

Pipeline: organizationId, name

Stage: pipelineId, name, order (sequence)

Relationships: Stage 1:N with Deal, used in DealStageHistory

Deal

Purpose: Track sales opportunities through pipeline

  • Amount: value (Decimal), probability (0-100)
  • Status: OPEN | WON | LOST
  • Timeline: expectedClose (DateTime)
  • Processing: isProcessed, isArchived

Relationships: N:1 with Pipeline/Stage/Contact, 1:N with Activity/DealStageHistory

DealStageHistory

Purpose: Audit trail of deal movements through stages

  • Movement: fromStageId → toStageId
  • Audit: changedBy (userId), changedAt (timestamp)

Use Case: Generate sales velocity reports, forecast accuracy analysis

HRM Module - Human Resource Management

Employee

Purpose: Central employee record with full profile and relationships

  • Identification: employeeCode (unique per org)
  • User Link: userId (optional, connects to system user)
  • Structure: departmentId, designationId, teamId, locationId
  • Hierarchy: managerId (self-referential for manager relationship)
  • Employment: joinDate, confirmationDate, employmentType, workMode, status
  • Compensation: salary (Decimal), bankAccountNumber, bankIFSC, taxId
  • Personal: dateOfBirth, gender, phone, address

Relationships: 1:N with Attendance, LeaveBalance, Payslip, PerformanceReview

Department, Designation, Team, Location

Purpose: Define organizational structure hierarchy

Constraint: Unique (organizationId, name) for each

Relationships: 1:N with Employee

Shift & EmployeeShift

Purpose: Define and assign work schedules

Shift Fields: name, startTime, endTime, graceMinutes

Relationships: 1:N through EmployeeShift junction

Attendance

Purpose: Track daily attendance with time tracking

  • Time: date, checkIn, checkOut
  • Calculation: totalHours, overtimeHours
  • Status: isLate, status

Constraint: Unique (employeeId, date)

LeavePolicy & LeaveBalance

Purpose: Define and track leave entitlements

Policy: organizationId, name, annualQuota, carryForward

Balance: employeeId, leavePolicyId, balance

Constraint: Unique (employeeId, leavePolicyId)

SalaryComponent & SalaryStructure

Purpose: Define salary components and structure per employee

Component: name, type, isPercentage, value

Structure: employeeId, componentId (junction)

Inventory Module

Product

Purpose: Master product catalog

  • Identity: sku (unique per org), hsnCode
  • Pricing: unitPrice, costPrice, taxRate
  • Status: isActive

Relationships: 1:N with StockItem, StockMovement, PurchaseItem

Warehouse & StockItem

Purpose: Physical locations and inventory levels

Warehouse: name, location, latitude, longitude

StockItem: productId, warehouseId, quantity (tracked per warehouse)

Constraint: Unique (productId, warehouseId)

StockMovement

Purpose: Audit trail of inventory transactions

  • Type: PURCHASE | SALE | ADJUSTMENT | TRANSFER
  • Tracking: quantity, referenceId (link to source doc)

Use Case: Full traceability and variance analysis

Accounting & Finance Module

LedgerAccount

Purpose: Chart of accounts for double-entry bookkeeping

  • Type: ASSET | LIABILITY | EQUITY | INCOME | EXPENSE
  • Hierarchy: parentId (self-referential tree)
  • Code: code (unique per org)

Relationships: 1:N with JournalEntry

Transaction

Purpose: Core accounting transaction record

  • Type: SALE | PURCHASE | EXPENSE | PAYMENT | REFUND
  • Status: DRAFT | POSTED | PAID
  • Reversal: reversalOfId (self-referential)

Relationships: 1:N with JournalEntry, PaymentAllocation

JournalEntry

Purpose: Double-entry bookkeeping records

  • Amount: debit XOR credit (only one non-null)
  • Invariant: Sum of debits = Sum of credits per transaction

eCommerce Module

Store

Purpose: Customer-facing storefront

  • Identity: name, domain (unique)
  • Management: ownerUserId, isActive
  • Payment: UPI ID, bank account details

Relationships: 1:N with Category, ProductListing, Cart, Order

ProductListing

Purpose: Store-specific product variant

  • Pricing: sellingPrice
  • SEO: seoTitle, seoDescription, images[]
  • Status: isActive

Constraint: Unique (storeId, productId)

Cart & Order

Purpose: Shopping cart and order management

Cart: storeId, customerId (unique combo), items[]

Order: status, paymentStatus, totalAmount, fulfillment tracking

Fulfillment: packedAt, shippedAt, deliveredAt, trackingNumber

Invoice & GST Module

Invoice

Purpose: Tax invoice generation with GST compliance

  • Reference: invoiceNumber (unique per org), orderId, posOrderId
  • GST: cgstAmount, sgstAmount, igstAmount (separate tracking)
  • Status: ISSUED | CANCELLED

Relationships: 1:N with InvoiceItem, EWayBill, PaymentAllocation

EWayBill

Purpose: Electronic way bill for interstate transport

  • Transporter: transporterName, transporterGSTIN, vehicleNumber
  • Route: transportMode, distance
  • Status: GENERATED | CANCELLED

Multi-tenant Data Isolation Key Principles

Key Principles:
  • Every table contains organizationId field
  • All indexes include organizationId for query efficiency
  • Foreign key constraints prevent cross-organization access
  • Row-level security enforced at application layer
  • Query pattern: WHERE organizationId = ? AND ...

Data Flow

1. Subscription to Module Activation

  1. Organization creates Subscription (references planId)
  2. Plan includes multiple PlanModule entries
  3. Application enables corresponding OrganizationModule records
  4. User access determined by: role + module enabled status

2. Order to Invoice to Payment

  1. Customer creates Order with OrderItems
  2. Payment captured: Order status → PAID
  3. Invoice generated, linked to Order
  4. InvoiceItems created from OrderItems (with GST calculation)
  5. EWayBill generated if interstate shipping
  6. PaymentAllocation matches Payment to Invoice

3. Inventory Stock Movement

  1. PurchaseOrder created with PurchaseItems
  2. On receipt: StockMovement record created (type=PURCHASE)
  3. StockItem quantity updated per warehouse
  4. GatePass tracks physical transfer
  5. Complete traceability: Product → Warehouse → Quantity

4. HRM Payroll Process

  1. EmployeeShift defines working schedule
  2. Daily Attendance record created (checkIn/checkOut)
  3. LeaveRequest reduces LeaveBalance
  4. SalaryStructure + Payslip generated monthly
  5. Data flow: Employee → Shift → Attendance → Payslip → Payment

Indexing Strategy

Critical Indexes for Performance

Primary Indexing Strategy:

All multi-tenant tables indexed on organizationId as first field to ensure query performance and data isolation.

High-Priority Indexes

-- Multi-tenant Isolation
User.organizationId (FK)
Contact.organizationId (FK)
Product.organizationId (FK)
Employee.organizationId (FK)
Order.organizationId (FK)
Invoice.organizationId (FK)
Transaction.organizationId (FK)

-- High-Query-Volume Tables
Transaction(organizationId, transactionDate)
Invoice(organizationId, issuedAt)
Order(organizationId, customerId, status)
Attendance(employeeId, date) UNIQUE
StockMovement(organizationId, productId, warehouseId)

-- Junction/Linking (Unique Constraints)
OrganizationUser(organizationId, userId)
ProjectMember(projectId, userId)
StoreMember(storeId, userId)
EmployeeShift(employeeId, shiftId)

Query Optimization Recommendations

  • Always filter by organizationId first in WHERE clause
  • Use composite indexes on (organizationId, foreignKey) for joins
  • Consider table partitioning by organizationId (future optimization)
  • Archive transactions older than 2 years to improve query speed
  • Cache frequently accessed data: Plans, Modules, LedgerAccount hierarchy

Performance Metrics

  • Typical Transaction Volume: 10M+ rows in production
  • Query Complexity: Mostly O(1) lookups; O(n) for reporting
  • Caching Layer: Redis for Plans, Modules, Hierarchy
  • Batch Processing: Monthly reconciliation async jobs
  • Data Retention: 7-year compliance for financial records

Data Integrity Constraints

Enforced Rules:
  • Subscription status transitions: valid state machine enforced
  • GST Calculations: CGST + SGST ≥ IGST (for non-interstate)
  • Attendance Uniqueness: one record per (employeeId, date)
  • Stock Levels: StockItem.quantity ≥ 0 (no negative stock)
  • Leave Balance: cannot go negative
  • Invoice Uniqueness: invoiceNumber unique per organization

Compliance & Audit

  • GDPR: Data isolated by organizationId
  • India GST: Compliant Invoice + EWayBill generation
  • PII Protection: Encrypted storage for SSN, bank details
  • Audit Trail: DealStageHistory, StockMovement, LeaveRequest history
  • Data Retention: 7-year retention for financial records
